## 1. 产品概述

### 1.1 一句话定位

**云表是国内唯一以"画表格编程"为核心范式的无代码企业应用开发平台，独创填表公式+业务公式双引擎，可稳定支撑 MES/WMS/ERP/SRM 等工业级复杂应用的全生命周期开发与运行。**

### 1.2 核心设计哲学

云表的核心信念是：**"懂业务的人最应该开发业务软件"**。为此，它将编程范式彻底重塑为业务人员最熟悉的工具——电子表格。

- **"表格即编程"**：摒弃传统 IDE，以类 Excel 的表格作为数据模型定义、界面设计与业务逻辑编写的一体化媒介。
- **"所见即所得"**：表单模板绘制即数据库建表，界面即数据模型，设计态与运行态合一。
- **"无代码，非低代码"**：云表强调自己并非"低代码"（Low-Code），而是彻底的"零代码"（No-Code）——不要求使用者具备任何代码编写或数据库设计知识。
- **中文式开发**：业务逻辑配置使用纯中文界面和中文条件描述，将技术门槛降至业务人员级别。

### 1.3 团队与公司

| 项目 | 详情 |
|------|------|
| 公司名称 | 珠海乐图软件有限公司 |
| 英文品牌 | Eversheet |
| 中文品牌 | 云表 / 云表平台 / 云表企业应用PaaS平台 |
| 创始人 | 张军 (表格编程技术倡导者) |

| 技术起步 | 自主研发电子表格内核，不依赖 Excel 第三方软件 |

### 1.4 核心差异化

在国产低代码/零代码赛道中，云表占有独一无二的生态位：

| 维度 | 简道云 | 明道云 | 云表 |
|------|--------|--------|------|
| 核心理念 | 表单驱动 + BI | 业务对象建模 (APaaS) | 表格编程 + 双引擎公式 |
| 架构分类 | 组件编排型 | 组件编排型 | 模型驱动型 (元数据映射 + 自动代码生成) |

| 部署模式 | SaaS + 私有化 | SaaS + 私有化 | 本地化 + K8S + 混合云 |
| 最大特点 | 帆软BI基因 + 钉钉生态 | 协作与流程引擎 | **零代码开发工业核心系统** |

## 2. 技术架构深度分析

### 2.1 整体架构：客户端-服务器分层计算模型

云表的架构采用严格的 **C/S 分层模型**，将计算明确划分在客户端和服务器端两个区域：

```
┌──────────────────────────────────────────────────┐
│                    云表浏览器 (客户端)              │
│  ┌────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ 表单渲染   │  │ 填表公式引擎 │  │ UI 交互   │ │
│  │ (Excel式)  │  │ (20+事件类型) │  │ (拖拽/扫描)│ │
│  └────────────┘  └──────────────┘  └───────────┘ │
│           │              │               │        │
│           │   JSON 数据包 (主从表结构)     │        │
│           ▼              ▼               ▼        │
├──────────────────────────────────────────────────┤
│                    云表服务器                      │
│  ┌────────────┐  ┌──────────────┐  ┌───────────┐ │
│  │ 数据持久化 │  │ 业务公式引擎 │  │ 工作流引擎│ │
│  │ (MySQL/    │  │ (服务器端触发)│  │ (BPMN2.0) │ │
│  │  SQL Server│  │              │  │           │ │
│  │  /Oracle)  │  │              │  │           │ │
│  └────────────┘  └──────────────┘  └───────────┘ │
└──────────────────────────────────────────────────┘
```

**核心抽象：表单 = 主从数据表 = JSON 文件**

云表中，"表单"并非传统的 UI 概念，而是一个流转于客户端和服务器端的"主从数据表"对象：
- 客户端维护为 JSON 格式的二维数据集（主表 + 明细表）
- 服务器端对应关系型数据库中物理的主-从表对
- JSON 序列化在网络中传输，两端共享数据

### 2.2 底层数据模型

云表采用 **元数据驱动** 的模型驱动架构（Model-Driven）：

1. **模板 (Template)**：最低层抽象，定义了一个主-从数据表结构和对应的 UI 界面
2. **数据项 (Data Item)**：模板中的字段，支持文本、数值、日期、下拉列表、附件、条形码等多种类型
3. **数据接口 (Data Interface)**：对数据表的提炼与过滤，返回特定格式和范围的数据集
4. **总表 (Master Table)**：存储在数据库中的物理表，与模板一一映射

数据建模过程与传统开发的对比如下：

| 传统开发 | 云表方式 |
|----------|----------|
| 设计数据库表结构 (DDL) | 画表格定义模板字段 |
| 创建数据库表、索引 | 保存模板 → 自动建表 |
| 编写前后端 CRUD 代码 | 模板自带增删改查 |
| 写 SQL 关联查询 | 配置数据接口 (可视化) |
| 编写业务逻辑代码 | 配置业务公式 (中文向导) |

### 2.3 多数据库支持

云表不绑定单一数据库，支持：
- MySQL（最常用）
- SQL Server（企业级场景）
- Oracle（大型企业）
- 数据源抽象层允许在运行时切换底层数据库

### 2.4 部署架构

| 部署模式 | 说明 | 适用场景 |
|----------|------|----------|
| 本地私有部署 | 服务器部署在企业内网，数据完全自主 | 制造业、政府、合规要求高的企业 |
| 公有云部署 | 使用云表提供的阿里云服务器 | 中小企业快速启动 |
| 混合云部署 | 核心服务在私有云，边缘服务在公有云 | 多仓库协同、跨国业务 |
| K8S 容器化 | 支持 Kubernetes 集群编排 | 高可用、弹性伸缩场景 |

### 2.5 安全体系

- **等保认证**：已通过等保 2.0 三级认证
- **加密标准**：国密 SM4 加密算法，TLS 1.3 传输加密，AES-256 存储加密
- **权限体系**：管理权限 + 业务权限 + 操作权限 + 数据权限 四层权限矩阵
- **审计日志**：完整操作日志记录，支持 180 天回溯
- **组织模型**：支持多分支机构、矩阵型组织、部门/岗位/角色/人员/分工/职权/代理 等复杂组织架构
- **数据安全**：字段级加密存储，公有云仅存储密文

### 2.6 专有客户端

云表拥有自主研发的 **云表浏览器**，这是一个专用客户端应用：
- 基于 BS 架构（Browser-Server），但提供桌面应用的交互体验
- 内置填表公式引擎，在客户端完成交互式计算
- 支持离线操作，网络中断时不中断使用
- 自动生成移动端 APP，一次设计多端运行

## 3. 双引擎公式系统

云表最核心的技术创新在于其独创的 **双引擎公式系统**——填表公式和业务公式。这两套引擎运行在不同位置、处理不同层次的逻辑，协同工作形成完整的业务计算体系。

### 3.1 填表公式 (Client Formula)

**运行位置**：客户端（云表浏览器）
**核心作用**：响应用户交互，辅助填写表单，提升填写效率和规范性

**事件模型**：基于"事件驱动"的编程模型，云表提供了 **超过 20 种核心事件类型**，包括：

| 事件类别 | 事件示例 | 典型用途 |
|----------|----------|----------|
| 表单生命周期 | 新建打开表单后、保存前/后 | 初始化数据、自动填充 |
| 值变化事件 | 数据项值改变时 | 实时计算联动（数量×单价=金额）|
| 单击/双击事件 | 单元格、按钮单击 | 弹出选择框、触发操作 |
| 定时器事件 | 定时触发 | 自动刷新、轮询状态 |
| 键盘事件 | 按键触发 | 扫码枪输入、快捷键 |
| 硬件事件 | 外设信号触发 | PLC 状态变化、RFID 读取 |

**操作类型**：超过 50 种内置操作，包括：
- 数据操作：赋值、新建表单、打开表单、保存表单、删除表单
- 界面操作：光标定位到、打开URL、消息提示、询问对话框
- 数据源操作：调用数据接口、调用全局数据接口
- 硬件操作：摄像头扫码、上传文件、打印
- 工作流：启动流程、提交审批

**三层结构**：
```
填表公式
  ├── 公式的执行条件（逻辑表达式，true 才继续）
  │     e.g. "本表单.总金额 > 1000"
  ├── 数据源（为操作提供二维数据集）
  │     每行记录驱动一次操作执行（N 条记录 = 执行 N 次）
  └── 操作（具体的功能执行单元）
        └── 操作的执行条件（在数据源每条记录上判断）
```

### 3.2 业务公式 (Business Formula)

**运行位置**：服务器端（云表服务器）
**核心作用**：处理跨模板的后台业务逻辑，保证数据一致性和复杂的业务规则执行

**与填表公式的关键区别**：

| 维度 | 填表公式 | 业务公式 |
|------|----------|----------|
| 运行位置 | 客户端（云表浏览器） | 服务器端（云表服务器） |
| 数据交互方向 | 用其他模板数据填写**本模板** | 本表单变化对**其他模板**的后续影响 |
| 可见性 | 可逐步观察计算过程 | 一次性返回最终结果，中间不可见 |
| 执行能力 | 受限于用户电脑性能 | 服务器级计算能力 |
| 适用场景 | 用户交互优化、实时反馈、前端验证 | 数据一致性、业务规则执行、复杂计算 |
| 触发方式 | 事件驱动（表单生命周期） | 状态改变、保存表单后、网络请求 |

**设计哲学**："填表公式负责快体验，业务公式负责稳业务"

**业务公式的核心机制——"情况"（Condition）**：

业务公式的执行基于"情况"的判断。一个"情况"描述了客户端提交的数据集的某种状态：
- 系统先检查客户端当前是何种业务"情况"（例如：是新建出库单还是撤销出库单）
- 然后根据"情况"选择对应的操作模式（新建、更新、删除目标表）
- 每种"情况"可定义独立的子数据源扩展和执行条件

### 3.3 填表公式与业务公式的协同模式

以工厂打包装箱为例，展示双引擎协同工作的典型模式：

```
步骤1: 装箱员扫描条码
  ↓ (云表浏览器)
步骤2: 填表公式 响应扫描事件 → 实时显示商品信息、更新装箱清单界面
  ↓ (JSON 数据包提交)
步骤3: 业务公式 接收到保存请求 →
  ├── 情况1: 首次装箱 → 新建"装箱清单" + 更新"出库单"已出数量
  ├── 情况2: 再次装箱 → 更新"装箱清单"明细（同产品合并累加数量）
  └── 校验: 应出数量 ≤ 已出数量+1 → 否则拒绝操作
```

这种 **"客户端交互 + 服务器逻辑"** 的分层协同，使得装箱员只需扫描两次条码即可完成整个装箱业务，而无需在任何页面间跳转。

## 4. 数据接口与子数据源体系

### 4.1 数据接口 (Data Interface)

数据接口是云表中数据调用的标准化方式，它是对服务器端数据表的"提炼和过滤"：

**核心能力**：
- **多数据源关联**：不限于单表，可跨多个模板组合数据
- **自定义返回值**：指定返回哪些字段
- **聚合计算**：支持记录数统计、求和、平均值等聚合函数
- **排序与过滤**：类似 SQL 的 WHERE + ORDER BY，但通过可视化配置
- **参数化查询**：支持按参数（如年、月、仓库ID）过滤数据
- **权限与策略**：数据接口的执行受权限管理约束

**数据接口的应用场景**：
- 下拉选择框的数据源（填写规范）
- 填表公式的数据源（为操作提供数据）
- 业务公式的子数据源（扩展计算数据）
- 跨模板的数据调用

### 4.2 子数据源体系 (Sub-Data-Source)

子数据源是云表业务公式中极其强大的数据扩展机制，类似关系型数据库的嵌套 JOIN 查询，但以可视化、分层的循环驱动方式实现：

**子数据源的本质**：
```
子数据源 = 数据接口查询结果 + 逐行循环驱动 + 多级扩展
```

**扩展方式**：
- **横向扩展**：将基本表扩展为更"宽"的表（加列）
- **纵向扩展**：将基本表扩展为更"长"的表（加行）

**单级 vs 多级**：

```
单级子数据源（同级）:
  供应商列表 [3条，勾选"多记录"]
  仓库列表   [2条]
  → 结果: 3条记录 (仓库列表横向扩展供应商列表)

多级子数据源（父子级）:
  供应商列表 [3条，勾选"多记录"] — 父级
    └── 仓库列表 [2条，勾选"多记录"] — 子级
  → 结果: 3×2=6条记录 (子级既横向又纵向扩展父级)
```

**动态子数据源**：每次循环操作都可能修改子数据源的数据，使子数据源做到"动态扩展"效果，这是云表公式系统极其灵活的核心。

**驱动机制**："子数据源像心脏——每一条记录触发一次心跳，驱动后续操作的执行"

**设计组合约束**（源表驱动目标表的 4 种模式）：
1. 源表主表 → 目标主表 + 源表主表 → 目标明细
2. 源表主表 → 目标主表 + 源表明细 → 目标明细
3. 源表明细 → 目标主表 + 源表主表 → 目标明细
4. 源表明细 → 目标主表 + 源表明细 → 目标明细

## 5. 核心功能

### 5.1 数据建模

- **模板设计器**：类 Excel 的表格绘制界面，拖拽式字段定义
- **字段类型**：文本、数值、日期、时间、附件、图片、条形码/二维码、下拉列表、复选框、流水号
- **虚拟字段**：不存储物理数据，通过公式或数据接口动态计算
- **Excel 兼容**：支持 Excel 函数和公式语法（如 VLOOKUP、SUMIF）
- **数据规范**：字段级数据类型、默认值、必填、唯一性、数值范围约束
- **自动建表**：保存模板时自动在数据库中创建对应的主-从表（含索引）

### 5.2 界面设计

- **多面板分割**：独家专利技术，支持单界面内多表格布局
- **多标签页展示**：单个表单可分多个标签页组织字段
- **复杂交互**：按钮、超链接、命令按钮、状态按钮、组合按钮
- **移动端自适应**：一次设计，自动生成移动端 APP
- **打印模板**：可视化设计打印格式（装箱单、采购单等）

### 5.3 工作流引擎

- **标准支持**：BPMN 2.0 标准
- **流程节点**：审批、条件分支、子流程、会签/加签、回退
- **组织集成**：与组织模型深度融合，支持按部门/岗位/角色分配审批人
- **流程监控**：流程状态可视化跟踪、超时提醒

### 5.4 权限管理

四层权限矩阵：

| 权限层级 | 控制粒度 | 说明 |
|----------|----------|------|
| 管理权限 | 模块/菜单级 | 哪些人能访问哪些系统模块 |
| 业务权限 | 表单/操作级 | 哪些人能对哪些表单执行增删改查 |
| 操作权限 | 按钮/功能级 | 哪些人能使用打印、导出、审批等操作 |
| 数据权限 | 记录/字段级 | 哪些人能看到哪些数据行、哪些字段 |

### 5.5 IoT 与硬件集成

- **条码/二维码**：原生支持条码生成与扫描（扫码枪、PDA）
- **RFID**：支持 RFID 读写设备接入
- **PLC 连接**：工业自动化设备数据采集
- **摄像头**：拍照上传、证件识别
- **AGV 集成**：无人搬运车任务调度（已应用于客户柔性生产项目）

### 5.6 系统集成

- **API**：RESTful API 标准接口，支持与其他系统（SAP、金蝶、用友）对接
- **Webhook**：事件驱动的对外通知机制
- **Python 嵌入**：支持 Python 脚本深度定制（复杂算法、外部系统交互）
- **钉钉/企微**：与主流协同办公平台集成
- **数据库直连**：支持外部数据库连接和数据同步
- **Datafocus 集成**：与数据分析平台 Datafocus 对接，实现填报+分析一体化

### 5.7 应用商店

- **行业模板**：200+ 行业模板，覆盖 OA、CRM、进销存、WMS、MES、ERP、SRM、HRM

## 10. AUDEBase 可借鉴点

### 10.1 填表公式/业务公式双引擎 → AUDEBase 计算字段分层

云表的双引擎公式系统为 AUDEBase 的 Schema Engine 提供了极有价值的设计参考：

```
AUDEBase 可借鉴的映射：

云表填表公式 (客户端)  →  AUDEBase 前端计算字段
    - 值变化事件驱动
    - 实时 UI 联动 (数量×单价=金额)
    - 前端数据校验

云表业务公式 (服务器端) →  AUDEBase Schema Engine 服务端计算字段
    - 保存前/后触发
    - 跨表复杂计算
    - 事务性保证
    - 子数据源多级扩展 → AUDEBase 嵌套数据源查询
```

**具体借鉴点**：

1. **公式分层**：将计算公式明确定义为"前端计算"（即时响应）和"后端计算"（事务保证）两类，与 AUDEBase 的 Schema Engine 计算字段设计对应
2. **数据源抽象**：借鉴"数据接口"概念——将复杂的跨表查询封装为可命名的数据源，简化 Schema Engine 的关联字段声明
3. **循环驱动**：子数据源的逐行循环机制可参考为 AUDEBase 的批量操作触发器设计

### 10.2 "画表格"编程 → Schema Engine 的表单式建模 UX

云表最成功的产品设计是将数据库建模与 Excel 体验融合——业务人员无需理解"表"、"字段"、"关系"等概念，只需画表格。

**AUDEBase 可借鉴**：
- Schema Engine 的 UI 构建器可提供"类 Excel 视图"作为数据建模的替代入口
- 将字段定义与表单布局设计合并为一（云表的核心创新）
- 降低 AUDEBase 管理后台的数据建模门槛

### 10.3 三模式融合 → AUDEBase 扩展分层

云表的"零代码(画表格) + 低代码(公式配置) + 全代码(Python 嵌入)"三层扩展体系：

```
AUDEBase 可对应的扩展分层：

零代码层   →  管理 UI 的可视化配置 (角色、权限、菜单)
配置层     →  manifest.yaml 声明 + Schema JSON 定义
低代码层   →  插件开发 (TypeScript + Fastify)
全代码层   →  自定义插件 + 数据库直连 (db_direct: true)
```

### 10.6 云表最值得学习的架构决策

## 总结

云表是国内低代码/零代码市场中一个独特的存在。它用 **"画表格"** 这一朴素但深刻的产品理念，证明了零代码平台不仅能做简单的 OA 审批，还能支撑 MES/WMS/ERP 级别的工业核心系统。

对 AUDEBase 而言，云表最重要的启示有三：

1. **抽象的一致性**：将"表格"作为贯穿数据建模、UI 设计、业务计算的一体化抽象，是降低门槛的核心策略
2. **计算的分层**：填表公式（快体验）+ 业务公式（稳业务）的双引擎分层，是复杂业务系统的正确计算架构
3. **开源 vs 闭源**：云表的闭源是其最大战略脆弱性。AUDEBase 以 Apache 2.0 开源，结合插件生态 + 标准化格式，可从根本上消除供应商锁定风险

云表的成功证明了"懂业务的人应该开发业务软件"这一理念的商业可行性。AUDEBase 应以开源+标准化+可扩展的方式，将这一理念推向更广阔的开发者生态。
